# Review task: adversarially audit Multipaste v2.4.0

You are the **reviewer** of the Multipaste v2.4.0 change (paste-as-plain-text
+ snippet-trigger fix). Review it as if someone else wrote it; you have **no
stake in defending it**. This holds whether you are a fresh window OR the same
window continuing after a `/compact`: treat the change adversarially, assume
the author was competent but fallible, and assume any summary you were handed
is plausible but possibly wrong. Your job is to find what's wrong, what's
risky, and what's missing, then fix it to ship quality.

(If you are continuing this same window after a compaction: your first two
actions are `cd ~/code/multipaste && git checkout feature/plain-text-paste`
and re-reading `docs/reviews/v2.4.0-COMPACTION-BRIEF.md`, because the compacted
summary is lossy and the on-disk brief is authoritative.)

## 0. Orient (do this before forming any opinion)

Read `docs/reviews/v2.4.0-COMPACTION-BRIEF.md` in this repo for context, then
establish ground truth yourself; **do not trust the brief; verify it (CoVe).**

```sh
cd ~/code/multipaste && git checkout feature/plain-text-paste
git diff main...feature/plain-text-paste -- . ':(exclude)docs/IMPROVEMENT-ROADMAP.md'
make test                                          # confirm the 288 baseline is real and green
swift scripts/plaintext-paste-smoke-test.swift     # confirm the smoke test passes
```

Read the actual files the diff touches; `PasteFlavor.swift`, `Paster.swift`,
`PickerWindow.swift`, `AppDelegate.swift`, `HistoryStore.swift`,
`MultiPasteComposer.swift`, `Preferences.swift`, `SettingsWindowController.swift`,
and the new/changed tests. Every finding must cite `file:line` and be
reproducible from the code, not from the brief.

## 1. Reverse-prompt yourself, then answer against the code

Generate the sharpest questions that would expose a defect in THIS change,
then answer each by reading the code / running something. At minimum cover
these lenses (add your own):

- **Correctness / logic.** Is the flavor threaded through *every* paste path,
  or does a path still call `Paster.put` without a flavor? Trace the combined
  and sequential multi-paste cases with the pref both ON and OFF. Does
  `effectiveFlavor`'s Shift-inversion do the right thing in all four
  (pref × shift) combinations? Is `MultiPasteComposer.textRepresentation`
  truly behavior-preserving after being turned into a forwarder?
- **Security / data exposure.** This writes to the real system pasteboard.
  Does plain-text paste ever expose something the user didn't intend
  (e.g. a full filesystem path via the files→path-text mapping in a context
  where they expected the file)? Can the snippet-trigger *inheritance* be
  abused; could re-copying content cause an unexpected snippet expansion, or
  inherit a trigger onto an item that then fires where the user didn't mean
  it? Does the injectable `pasteboard` parameter open any misuse? Does
  stripping `.rtf` ever *reveal* hidden/obscured content that the rich form
  concealed? Are the nspasteboard privacy markers (concealed/transient) still
  honored on the paste-out path?
- **Edge cases.** Empty string; whitespace-only; very large text; multi-byte
  / emoji / RTL; an item whose RTF `plain` fallback is empty but rtf is
  non-empty; a file URL with spaces/newlines in the path; a mixed
  marked-multi-paste (text + files + image) pasted `⇧↩`; the `.clipboardOnly`
  route (no target app) with a plain flavor; double-click vs `⇧`-double-click.
- **Thread-safety / races.** `HistoryStore` declares a `DispatchQueue` that is
  never used (dead code) and mutates `items` on the main thread only; is that
  actually guaranteed given the poll timer and picker both touch it? Does the
  trigger-inheritance change interact with dedup/eviction ordering?
- **API / type design.** Is `PasteWrite` the right seam? Is the `.rich`
  default on `Paster.put` a footgun (silent rich paste if a caller forgets a
  flavor)? Should `PlainText.string` and `pasteWrite` handle any kind
  differently? Is `Equatable` on `PasteWrite` sound for `Data`/`[URL]`?
- **Backward compatibility.** Is the `.rich` write byte-for-byte identical to
  pre-2.4.0 (same declared types, same order, same data)? Prove it, don't
  assume. Do old `history.json` / preference plists still load unchanged?
- **Performance.** Any new per-paste allocation or copy that matters? (Low
  stakes here, but check.)
- **Test coverage gaps + mutation survival.** What mutation to the new code
  would the 288-test suite NOT catch? For each gap you find, add the missing
  test. Verify existing new tests have teeth (inject the bug, confirm exactly
  the right test reddens, revert).
- **macOS pasteboard pitfalls.** `clearContents` + `setString` without
  `declareTypes` for the plain path; is that correct on macOS 13? Does the
  `ClipboardMonitor` re-capture (changeCount) behave for a plain paste? Any
  interaction with `PasteboardAugmenter`'s file-path injection?
- **Docs + a11y.** Does every doc claim (README keys/settings/coverage table,
  CHANGELOG, the `⇧↩` hint) match the code exactly? Is the new checkbox /
  hint accessible (VoiceOver label, contrast)?

## 2. Verify like the author's CLAUDE.md demands (CoVe)

For every load-bearing claim you make; "this is a bug", "this leaks", "this
is safe", "this test is green"; write the check that proves it and RUN it
(read the exit code; re-run `make test`; run the smoke script; trace the
code path). Do not assert from plausibility. If you can't verify a suspicion,
label it "unverified" and say what check would settle it. Watch for the
false-positive trap: confirm a suspected bug actually reproduces before
reporting it.

## 3. Deliverable (the finished product, not a plan)

1. A prioritized **findings report** (`docs/reviews/v2.4.0-FABLE5-FINDINGS.md`),
   most-severe first, each with: severity, `file:line`, the concrete failure
   scenario (inputs → wrong output), and whether you confirmed it reproduces.
2. **Fix every confirmed defect** on a new branch off `feature/plain-text-paste`
   (e.g. `review/v2.4.0-fable5`), each fix with a regression test in the
   existing harness style, **mutation-tested** (inject the bug, confirm
   exactly your new test reddens, revert). Keep the `.rich` path
   behavior-preserving.
3. Re-run `make test` + `make plaintext-smoke-test` + `make verify-app` and
   report the real counts / exit codes. Update CHANGELOG/README/version only
   if a fix warrants it, keeping every `VersionConsistency`-guarded artifact
   in sync.
4. If you find NOTHING wrong after a genuine adversarial pass, say so plainly
   and show the checks that justify the all-clear; don't invent findings to
   look busy, and don't rubber-stamp to look agreeable.

## Standard

Boil the ocean. The marginal cost of completeness is near zero. Do it right,
with tests and documentation, so well that Rohin is genuinely impressed. Never
table a thing when the permanent solve is within reach; never leave a dangling
thread; never present a workaround when the real fix exists. The bar is not
"good enough"; it's "holy shit, that's done." Extract and document anything
reusable you produce (a new test pattern, a reusable check) so a future
session benefits.
