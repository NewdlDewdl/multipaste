## Fixed defects (most severe first)

### F1 (medium, security): snippet trigger could silently rebind to foreign bytes
- **Where:** `Sources/MultipasteCore/HistoryStore.swift` `insert` (the
  v2.4.0 trigger-inheritance line), with `ClipboardItem.swift:109`
  (`contentHash = "rtf:" + sha256(plain)`).
- **Failure:** a rich-text hash covers only the PLAIN text, so any clip
  with identical visible text but different RTF bytes (fonts, colors, an
  injected hyperlink) collides with an existing snippet. The v2.4.0
  inheritance then attached the user's trigger to the incoming payload:
  typing `;sig` would expand bytes the user never approved, invisibly
  (SnippetEngine pastes rich). Confirmed with an executed repro against
  the real `MultipasteCore` build. Pre-2.4.0 the same collision merely
  killed the trigger; the original fix accidentally upgraded a
  denial-of-snippet into a payload swap.
- **Fix:** a re-copy that collides with a snippet (existing item has a
  trigger, incoming does not) resurfaces the EXISTING item wholesale:
  payload, pin, trigger, id. Non-snippet re-copies keep adopt-newest
  semantics; an incoming item with its own trigger still wins (explicit
  user action).
- **Tests:** `HistoryStore/reCopySamePlainDifferentRtfKeepsSnippetPayload`,
  `HistoryStore/reCopyNonSnippetAdoptsNewestPayload` (+ the two existing
  reCopy tests still pass). **Mutation-proven:** reverting to
  adopt-incoming reddens exactly the payload test (1 failed, 301 passed).

### F2 (medium): `â‡§â†©` on an empty-plain rich clip clobbered the clipboard with nothing
- **Where:** `Sources/MultipasteCore/PasteFlavor.swift` `pasteWrite`.
- **Failure:** an RTF stub like `{\rtf1\ansi}` parses to an empty string
  and IS capturable (`ClipboardMonitor.snapshot` guards emptiness only on
  the plain-text branch; verified by parsing the stub with
  `NSAttributedString`, `attr.string.isEmpty == true`). `â‡§â†©` then wrote
  `.string("")`: clipboard cleared, nothing pasted, previous clipboard
  content destroyed.
- **Fix:** an empty plain form falls back to the rich write, exactly like
  an image with no plain form. Strictly-empty only: whitespace-only plain
  text still pastes plain (pasting spaces/newlines is legitimate), and
  `.text("")` is unchanged (its rich write is `.string("")` too).
- **Tests:** `PlainText/pasteWritePlainEmptyRtfFallsBackToRich`,
  `pasteWritePlainWhitespaceRtfStaysString`,
  `pasteWritePlainEmptyTextIsEmptyString`; smoke check 5 (script) and
  paste-smoke check 5 (real executor) prove it on live pasteboards.
  **Mutation-proven:** removing the guard reddens exactly the fallback
  test; over-tightening it to trim whitespace reddens exactly the
  whitespace test.

## Accepted as designed (documented, not changed)

- **Plain paste re-captures as a new history entry.** `â‡§â†©` on a rich item
  writes plain text; the monitor's self-capture (deliberate since v2.3.0
  for merged multi-pastes) inserts it as a NEW item (different
  contentHash prefix), so history gains a reusable plain copy and row 0
  changes. Now documented in README + CHANGELOG as behavior, with the
  rationale. Marking our own writes transient would break the documented
  multi-paste-lands-in-history feature.
- **Files pasted plain expose full paths.** Identical to the existing
  path-augmentation text for single file copies; deliberate.
- **`â‡§â†©` on an image pastes the image.** Deliberate fallback, documented.
- **RTF capture derives `plain` by parsing the RTF** rather than reading
  the source app's own `.string` representation (pre-existing in
  `ClipboardMonitor.snapshot`, untouched by v2.4.0). A source app's
  `.string` can differ from the parse; plain paste then emits the
  approximation. Low impact, capture-side, and changing it alters
  contentHash/dedup semantics: left for its own change with its own smoke
  coverage. Roadmap item.
