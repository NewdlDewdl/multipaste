## 2. What changed and why

### Feature: paste as plain text (`⇧↩`)

**Problem (pre-2.4.0):** `Paster.put` wrote both `.rtf` and `.string` for a
rich-text item, so a styled clip (webpage / Word / Notion) dragged fonts,
colors, and sizes into plain fields, code editors, and chat boxes, with no
keyboard escape.

**Design (verify this is what the code does):** the pasteboard-write
*decision* was moved into a pure, unit-tested Core policy; the app layer only
executes it.

- **`Sources/MultipasteCore/PasteFlavor.swift` (NEW):**
  - `enum PasteFlavor { case rich, plainText }`
  - `enum PasteWrite: Equatable { case string(String); case richText(rtf: Data, plain: String); case image(png: Data); case fileURLs([URL]) }` — a full description of what to put on the pasteboard.
  - `PlainText.string(for: ClipboardItem) -> String?` — the unstyled string per kind: `.text` → verbatim; `.rtf` → the stored plain fallback (NOT the rtf bytes); `.fileURLs` → `PasteboardAugmenter.pathText(forFiles:)`; `.image` → `nil`.
  - `PlainText.pasteWrite(for:flavor:) -> PasteWrite` — `.rich` reproduces the old per-kind write exactly; `.plainText` collapses text-bearing kinds to `.string(plain)` and falls back to the rich write when there's no plain form (images).
- **`MultiPasteComposer.textRepresentation`** now forwards to `PlainText.string` (one source of truth; a test asserts they agree).
- **`Preferences.plainTextPasteDefault`** (Bool, default **false**).
- **`Sources/Multipaste/Paster.swift`:** `put(_ item:, flavor: PasteFlavor = .rich, to pasteboard: NSPasteboard = .general)` is now a thin executor over `PlainText.pasteWrite`. The `.rich` default means every existing caller (snippet expansion via `SnippetEngine`, menu quick-pick, ⌘1-9) is unchanged. `pasteboard` is injectable for the smoke test.
- **`Sources/Multipaste/PickerWindow.swift`:** `effectiveFlavor(shiftPressed:)` = base (the pref) inverted by Shift. `onPick` widened to `([ClipboardItem], NSRunningApplication?, PasteFlavor)`. The Return branch (keyCodes 36/76) calls `commit(flavor: effectiveFlavor(shiftPressed: isShift))`; `commitSelection` (double-click) reads Shift from `NSApp.currentEvent`; ⌘1-9 uses `effectiveFlavor(shiftPressed: false)`. Hint legend shows `⇧↩ plain text`.
- **`Sources/Multipaste/AppDelegate.swift`:** the flavor threads through `pickAndPaste(_:previousApp:flavor:)` → `deliver` / `deliverSequentially` / `pasteSequentially` (ALL three call `Paster.put`, so all three were threaded).
- **`Sources/Multipaste/SettingsWindowController.swift`:** "Paste as plain text by default" checkbox in the General tab (writes the pref; the picker reads it live per pick, so no callback plumbing).

### Bugfix: re-copying a snippet's text silently killed the snippet

`HistoryStore.insert` preserved a re-copied item's `pinned` but not its
`trigger`. A fresh factory item has `trigger == nil`, and `SnippetMatcher`
only fires on a pinned item with a non-empty trigger, so re-copying a
snippet's exact body left a pinned-but-dead item. Fix: `insert` now does
`if fresh.trigger == nil { fresh.trigger = existing.trigger }` (inherits the
trigger the same way it already inherits `pinned`; incoming explicit trigger
wins).

## 3. Deliberate decisions (so the reviewer doesn't "fix" them as bugs)

- **Default OFF** so existing muscle memory (`↩` = paste what I copied) is unchanged; plain is opt-in via `⇧↩`. With the pref ON, `↩`/`⇧↩` invert.
- **⌘1-9 and ⌘-click** stay on the base flavor; Shift is scoped to the Return / double-click path only.
- **`⇧↩` on an image** falls back to pasting the image (no plain form) rather than a no-op.
- **Combined multi-paste:** all-text combine is already `.text`, so plain==rich; all-files combine in `.plainText` pastes the path text (consistent with a single file copy pasted into a text field).
- **Snippet expansion is unaffected** (it calls `Paster.put(item)` → default `.rich`).
- **Compare table in README got NO plain-text-paste row** — deliberately: competitor plain-text-paste support wasn't verified, and asserting unverified claims in a public artifact was refused (CoVe).
